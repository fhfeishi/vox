import re

VOICE_PATTERN = re.compile(r"请(?:你)?用【(.+?)】(?:的声音|声音)?(?:来)?回答")

VOICE_MAP = {
    "我": "speaker",           # 克隆说话人
    "默认": "default",         # 默认声音
    # 其余 key 直接映射 /locals/ref/{key}.wav

def parse_voice_command(text: str) -> tuple[str, str]:
    """返回 (voice_mode, cleaned_text)"""
    m = VOICE_PATTERN.search(text)
    if m:
        name = m.group(1).strip()
        mode = VOICE_MAP.get(name, name)  # 未命中则视为文件名
        clean = VOICE_PATTERN.sub("", text).strip()
        return mode, clean
    return "default", text

### 一、 核心交互逻辑优化（音色切换与指令系统）

你提到的通过正则 `re` 提取【】内的文字是一个很好的起点，但在真实语音对话中，ASR可能会识别错标点或字词（比如识别成“请用我自己的声音”、“换成张三的音色”等），**纯正则太脆弱了**。

**💡 优化建议：将“指令解析”交给 LLM（IIM/LLM API），采用结构化输出。**

1.  **指令解析机制**：
    *   在 System Prompt 中赋予 LLM 识别音色切换意图的能力，要求其返回结构化数据（如 JSON）。
    *   **System Prompt 示例**：*“你是一个语音助手。当用户要求切换声音时，请提取目标音色。返回格式：{"action": "switch_voice", "voice_target": "目标音色", "reply": "好的，已切换..."}。如果正常提问，返回：{"action": "answer", "reply": "回答内容”}”*
    *   **优势**：容错率极高。无论用户说“请用我的声音”、“换成我”、“用【我】的声音”，LLM都能提取出 `voice_target: "我"`。

2.  **三种音色模式的精准实现**：
    *   **默认模式**：启动时加载 `/locals/ref/default.wav`，全局维持该音色。
    *   **克隆说话人（我）**：**这正是你提到的“音色缓存机制”大显身手的地方！** 每次用户说话时，后端顺手截取用户一句高品质音频（比如3-5秒），存入内存或临时目录作为 `prompt_audio`。当LLM解析出 `voice_target: "我"` 时，TTS直接拿这句刚缓存的音频作为参考音色生成。

### 二、 实时性与可交互性（打断机制）保障

打电话最核心的体验是**低延迟**和**能打断**。这是整个产品技术成败的关键。

**💡 优化建议：流式处理 + VAD（语音活动检测）打断。**

1.  **极致流式 Pipeline**：
    *   传统的顺序：听完 -> 识别完 -> 生成完 -> 语音合成完 -> 播放（延迟 5-10秒，完全不能接受）。
    *   **正确的流式**：ASR流式输出 -> LLM流式输出（按标点切词） -> TTS流式合成 -> 前端流式播放。
    *   *具体做法*：LLM吐出“你好，”就立刻送给TTS合成，不用等后面的“我是助手”。这样延迟能压缩到 1-1.5秒以内。

2.  **用户打断机制**：
    *   必须引入 **VAD（Voice Activity Detection）**。推荐使用 `Silero VAD`（极小、极快、极准）。
    *   **打断逻辑**：当助手正在播放TTS音频时，如果VAD检测到用户开始说话（且持续超过300ms，避免咳嗽误触），**立刻执行**：
        1.  中止当前TTS音频播放。
        2.  清空LLM尚未生成的队列。
        3.  开始接收用户新的ASR输入。

        self.current_voice_ref = "defaults/default.wav" # 默认音色
        self.speaker_voice_cache = None                 # 说话人音色缓存
        self.is_speaking = False                        # 助手是否在说话

    async def handle_user_audio_stream(self, audio_stream):
        async for chunk in audio_stream:
            # 1. VAD 检测
            if vad_is_user_speaking(chunk):
                if self.is_speaking:
                    await self.interrupt_assistant() # 触发打断！清空TTS队列
                # 2. 缓存用户音频（用于“克隆说话人”）
                self.cache_speaker_voice(chunk) 
                # 3. 送入 ASR
                text = await asr_stream.recognize(chunk)
                if text:
                    await self.process_text(text)

    async def process_text(self, text):
        # 4. LLM 结构化输出 (意图识别 + 生成回复)
        result = await llm_api.parse_intent_and_reply(text)
        
        # 5. 音色切换判断
        if result.action == "switch_voice":
            if result.voice_target == "我":
                self.current_voice_ref = self.speaker_voice_cache
            elif result.voice_target in voice_mapping:
                self.current_voice_ref = f"/locals/ref/{voice_mapping[result.voice_target]}"
        
        # 6. 流式 TTS 合成与播放
        self.is_speaking = True
        async for text_segment in split_by_punctuation(result.reply): # 按标点切分流式送TTS
            audio_chunk = await tts_api.stream_synthesize(text_segment, ref_audio=self.current_voice_ref)
            await websocket.send(audio_chunk) # 直推前端
        self.is_speaking = False

# VOX 实时语音通话助手 - 产品需求文档 v2.0

## 一、产品概述与核心价值

### 1.1 产品定位

VOX 是一款基于语音大模型的实时通话助手，旨在为用户提供流畅、自然的语音交互体验。产品核心价值在于打破传统语音助手“你问我答”的机械式交互，真正实现**“类人与人通话”**的沉浸式对话体验。用户可以随时打断、插话、切换音色，系统能够快速响应并保持对话连贯性，真正模拟人类交流的自然节奏。

### 1.2 目标用户画像

本产品的目标用户主要包括以下几类：需要快速获取信息但不便手动输入的企业人士；追求个性化交互体验的科技爱好者；需要多语言翻译或跨文化交流的跨境工作者；以及希望获得情感陪伴的老年用户。理解目标用户的使用场景和痛点，将帮助我们在功能设计上做出更合理的取舍，确保产品能够在核心场景下提供极致的用户体验。

### 1.3 核心竞争优势

与市场上现有的语音助手相比，VOX 的核心竞争力体现在三个维度。首先是**毫秒级响应**，通过优化 Pipeline 架构和引入流式处理技术，实现端到端延迟低于 800ms 的行业领先水平。其次是**自然打断机制**，用户可以像与真人对话一样随时插话打断，系统会立即停止当前生成并响应新输入。最后是**灵活的音色切换**，支持实时克隆说话人声音或切换到预设音色，用户只需一句简单的指令即可完成切换，无需复杂配置。

## 二、功能详细设计

### 2.1 核心功能模块

#### 2.1.1 语音识别模块（ASR）

语音识别是整个 Pipeline 的入口，负责将用户的语音信号转换为文本。我们建议采用阿里云语音识别服务，其在中文场景下的识别准确率可达 98% 以上，且支持流式识别能够显著降低延迟。阿里云 ASR 提供以下核心能力：实时语音转文本，支持最长 5 小时的连续识别；说话人分离功能，能够区分不同说话人的声音；热词优化，允许用户上传专业领域词汇以提升识别准确率；标点智能预测，自动为识别结果添加标点符号。

在技术实现层面，ASR 模块需要支持 VAD（Voice Activity Detection，语音活动检测）功能，用于判断用户是否结束发言。VAD 的灵敏度直接影响用户体验：过于敏感会导致系统误判用户已说完话而过早生成回复；过于迟钝则会使用户感到等待时间过长。建议采用自适应 VAD 策略，根据用户最近的对话节奏动态调整判断阈值。此外，还需要实现噪音过滤和回声消除，确保在各种环境下都能准确识别用户语音。

#### 2.1.2 大语言模型模块（LLM）

LLM 是产品的“大脑”，负责理解用户意图并生成有价值的回复。考虑到产品的实时性要求，我们建议采用流式输出（Streaming）模式，让模型能够边思考边输出。流式输出可以将首字延迟从传统的 2-3 秒降低到 0.5 秒以内，大幅提升用户体验。模型选择上，建议使用国内主流大模型平台（如阿里云百炼），因为语音交互场景对响应速度和网络稳定性要求较高，本地化部署能够提供更好的延迟保障和合规性支持。

LLM 模块需要处理两类输入：纯文本对话和包含上下文的多轮对话。对于多轮对话，系统需要维护对话历史上下文，并考虑输入 token 数量限制下的上下文截断策略。建议采用滑动窗口方式保留最近 10-15 轮对话，既能保证对话连贯性又能控制 token 消耗。另外，LLM 需要能够识别用户切换音色的指令（如“请用张三的声音回答我”），并从指令中提取目标音色信息传递给下游 TTS 模块。

#### 2.1.3 语音合成模块（TTS）

语音合成是 Pipeline 的出口，负责将 LLM 生成的文本回复转换为语音输出。TTS 技术选择对最终用户体验影响重大。当前主流的 TTS 服务包括阿里云语音合成、讯飞语音云、Azure 语音服务等，各有优劣。阿里云语音合成的优势在于中文发音自然度高、支持多种音色、提供语音定制服务；讯飞在情感语音方面表现突出；Azure 语音服务则在国际化和多语言支持方面领先。

针对用户提出的多种语音克隆需求，我们建议采用**本地语音克隆 + 云端合成**的混合架构。具体方案如下：对于默认音色和预留音色，可以直接使用云端 TTS 服务提供的标准音色或预先定制好的音色，响应速度快且稳定性好；对于需要克隆特定人物声音的场景，可以先在本地使用开源语音克隆工具（如 Coqui TTS、XTTS）提取参考音频的声音特征，然后使用提取的声学特征驱动云端 TTS 或本地合成引擎。这种方案兼顾了响应速度和音色质量，同时降低了本地算力要求。

#### 2.1.4 语音克隆模块

语音克隆是本产品的核心差异化功能，需要精心设计以平衡用户体验和技术实现复杂度。我们设计了三种音色模式：

**默认模式**使用产品预设的音色，通常选择音色甜美/浑厚、语速适中的专业配音员声音。预设音色经过精心调优，在各种内容类型上都有稳定表现，适合大多数使用场景。用户首次使用时系统默认采用此模式，无需任何额外操作。

**自克隆模式**允许系统克隆当前说话人的声音，实现“用自己的声音与自己对话”的有趣体验。技术实现上，当用户首次使用此模式时，系统会提示用户朗读一段引导文本（约 30 秒），用于提取声音特征并存储到本地。后续使用时无需再次录音，声音特征会持久保存在用户配置文件中。当然，用户也可以随时重新录制引导文本以更新声音特征。

**指定克隆模式**支持克隆预设的特定人物声音或用户上传的参考音频。预设人物可以是名人、动漫角色、产品吉祥物等，每个预设人物对应本地存储的一段参考音频和对应的声音特征文件。用户通过指令“请用【角色名】的声音回答我”即可切换到对应音色。如果用户需要克隆自己上传的音频中的人物声音，只需说“请用我上传的【文件名】的声音回答我”，系统会自动在本地参考音频库中匹配。

### 2.2 语音模式切换机制

为了实现平滑、可靠的多音色切换，我们设计了基于自然语言指令的切换机制。用户可以通过以下方式切换音色：

| 切换方式 | 示例指令 | 系统行为 |
|---------|---------|---------|
| 切换到默认音色 | “请用默认声音回答我” | 使用产品预设音色 |
| 自克隆 | “请用我的声音回答我”或“请用我自己的声音回答” | 使用已存储的自克隆音色 |
| 切换到预设角色 | “请用小明的角色回答我” | 加载本地预设角色“小明”的音色 |
| 上传音频克隆 | “请用我上传的张三的声音回答” | 加载用户上传的张三参考音频的音色 |

指令识别采用正则表达式匹配 + LLM 理解的双重机制。正则表达式用于快速识别标准指令格式（如“请用【】的声音回答我”），LLM 则负责处理更自然、口语化的表达方式（如“别用那个声音了，换个正常的”）。这种双重机制既保证了切换指令的快速响应，又能处理各种可能的自然语言表达。

### 2.3 实时性保障机制

实时语音对话对延迟极为敏感，心理学研究表明，当系统响应延迟超过 300ms 时，用户会开始感受到明显的等待感；当延迟超过 1 秒时，对话的自然流畅感会被严重破坏。我们设计了一套多层次的延迟优化机制：

**流式 Pipeline 设计**：整个处理流程采用流式架构，ASR 识别到一定数量的文字后即开始发送给 LLM，LLM 生成文本时同步流式发送给 TTS，TTS 边接收边合成语音输出。这种流水线式的处理方式使得各环节可以并行工作，最大化利用等待时间。

**预测性响应优化**：系统会分析用户的历史对话模式，学习用户的提问习惯。当检测到用户可能提出某类常见问题时，可以提前启动响应生成，从而进一步缩短感知延迟。这种机制对于天气查询、时间询问等高频场景特别有效。

**音色特征缓存**：这是保证音色切换实时性的关键技术。每次音色切换时，系统需要加载声音特征数据。对于用户经常使用的音色，系统会将其特征数据缓存在内存中，使得切换操作可以在 100ms 内完成。只有对于长期未使用的音色，才需要从磁盘加载。对于自克隆音色，建议始终保持内存缓存以保证最佳体验。

**语音片段预合成**：对于一些高频使用的固定回复（如“好的”、“稍等一下”、“这个问题我帮你查一下”），可以预先合成对应的语音片段。当 LLM 输出包含这些片段时，直接拼接预合成音频而非实时合成，可以显著降低首字延迟。

### 2.4 打断机制设计

允许用户随时打断是实现“类人与人对话”体验的关键。打断机制的实现面临几个技术挑战：如何准确检测用户开始新输入、如何快速中止当前正在进行的语音合成、如何在打断后正确切换到新的对话上下文。

**打断检测**：当用户在 TTS 播报过程中开始说话时，ASR 模块需要能够检测到新的语音输入。这需要 ASR 支持双工模式或采用 VAD + ASR 双重监听。一种可行的方案是持续运行 VAD 检测麦克风输入，当检测到语音活动时立即触发打断流程。

**中断响应**：一旦检测到用户打断，系统需要立即执行以下操作——停止 TTS 播放（如果有缓冲音频则清空缓冲区）、中止 LLM 继续生成（发送停止信号）、将用户的打断语音发送给 ASR 进行识别。整个中断响应过程需要在 200ms 内完成，否则用户会感到明显的滞后。

**上下文处理**：打断后的上下文处理需要特别小心。如果用户在 LLM 生成回复的过程中打断，之前的输入文本可能只有部分被 LLM 处理。这时需要决定是丢弃之前的上下文重新开始，还是等待打断语音识别完成后将两部分拼接。建议采用后者，将打断前后的语音拼接为一个完整的输入发送给 LLM，以保证对话的连贯性。

## 三、用户交互设计

### 3.1 开机引导流程

产品启动时的引导流程直接影响用户的第一印象。我们设计了以下开机流程：

**阶段一：系统就绪提示**。当所有服务模块（ASR、LLM、TTS）完成初始化后，系统播放准备就绪提示音。提示音应简洁友好，控制在 5 秒以内，建议文案为：“你好，我是 VOX 语音助手，有什么可以帮你的？”同时，界面应展示简洁的状态指示器，让用户直观了解系统当前状态。

**阶段二：功能引导（首次使用）**。对于首次使用的用户，在提示音播放完毕后，系统可以简短介绍主要功能：“你可以随时打断我，也可以说'请用我的声音回答我'来使用你自己的音色。有什么想问的吗？”这种渐进式引导让新用户在短时间内了解核心功能，同时保持对话的自然感。

**阶段三：就绪等待**。引导完成后，系统进入等待状态。当检测到用户开始说话时（VAD 触发），立即开始语音识别和对话处理，Pipeline 全速运转。

### 3.2 界面设计规范

#### 3.2.1 整体布局

界面采用左主右辅的经典布局，左侧为对话主窗格（约 70% 宽度），右侧为功能面板（约 30% 宽度）。整体设计遵循“留白优先、信息密度适中”的原则，避免界面过于拥挤影响视觉体验和操作效率。

**主色调选择**：推荐使用深色主题，理由如下——语音交互场景下用户的注意力集中在“听”而非“看”，深色背景可以降低视觉疲劳；深色主题在技术产品中往往传递专业、高端的品牌调性；深色界面上的高亮元素（如波形、状态指示）更容易吸引用户注意。配色方案建议：背景色采用深灰渐变（#121212 → #1E1E1E），强调色使用科技蓝（#00D4FF）或活力橙（#FF6B35）点缀。

#### 3.2.2 对话窗格设计

对话窗格是用户与产品交互的核心区域，设计上追求“简约而不简单”的灵动美感。

**消息气泡**：采用圆角卡片设计，左右分布区分用户输入和产品输出。气泡背景使用微透明效果（如 rgba(255,255,255,0.05)），边框使用 1px 细线，视觉上轻盈通透。用户消息右对齐，使用者强调色；助手消息左对齐，使用辅助色或中性灰。

**语音波形可视化**：在对话窗格底部或侧边展示实时的语音波形，直观呈现 ASR 识别状态和 TTS 播放状态。波形颜色随状态变化：麦克风激活时为绿色、TTS 播放时为蓝色、等待响应时为灰色脉动。这种动态可视化让用户始终了解系统当前状态。

**时间戳与角色标签**：每条消息底部显示时间戳（新消息 5 分钟内不显示，超过则显示具体时间），消息上方显示角色标签（如“你的声音”、“小明的角色”），帮助用户明确当前使用的是哪种音色模式。

#### 3.2.3 功能面板设计

右侧功能面板承载产品使用说明和功能配置入口，设计上强调“触手可及但不喧宾夺主”。

**音色模式指示器**：面板顶部设置当前音色模式的视觉指示器，使用图标 + 文字的方式清晰展示当前使用的是哪种音色。点击指示器可展开音色切换选项，用户可以预览每种预设音色的声音样本。

**指令提示卡片**：面板中部放置简洁的指令提示卡片，采用问号图标引导。卡片内容示例：“切换音色？试试说——请用【】的声音回答我”，其中【】位置可替换显示可用的音色名称列表。卡片采用可折叠设计，不使用时收缩为一个小图标，保持界面简洁。

**使用技巧折叠区**：面板底部提供使用技巧的折叠区域，包含打断技巧（“随时打断我，我可以立即响应”）、多轮对话技巧（“我会记住对话历史”）、个性化技巧（“我可以学习你的声音”）等简短提示。这些技巧以图标 + 短句的形式展示，点击可展开详细说明。

#### 3.2.4 状态指示系统

建立清晰的状态指示系统，让用户始终了解系统运行状态：

| 状态 | 视觉表现 | 音效提示 |
|------|----------|----------|
| 空闲等待 | 麦克风图标静止，绿色呼吸灯 | 无 |
| 正在聆听 | 麦克风图标高亮，波形跳动 | 轻微提示音 |
| 处理中 | 加载动画，旋转图标 | 无 |
| 正在回答 | 波形动态播放，消息气泡出现 | TTS 语音输出 |
| 被打断 | 波形闪烁，波浪形指示 | 无 |
| 错误 | 红色警告图标 | 错误提示音 |

### 3.3 多模态交互暗示

除了视觉暗示，系统还通过语音和音效传递状态信息：

**状态切换音效**：设计一套统一的音效系统，用于传达不同状态切换。开始聆听时播放轻微的“滴”声；开始回答时播放上升音调的“叮”声；对话结束时播放轻柔的“嗖”声。这些音效音量适中、时长极短（建议 100-300ms），既不干扰对话又提供有效的状态反馈。

**音色切换提示**：当用户成功切换音色后，系统除了在界面上更新指示器，还应通过语音播报确认：“好的，现在用小明的声音回答你”。这样用户即使不看屏幕也能确认指令已被正确执行。

### 4.2 核心模块职责

#### 4.2.1 Pipeline 编排器（pipeline.py）

Pipeline 是整个系统的核心调度单元，负责协调 ASR、LLM、TTS 三个模块的工作流程。采用异步编程模式，基于 Python 的 asyncio 库实现高并发处理。

**Pipeline 的核心流程**包括以下步骤：首先接收用户语音输入并触发 ASR 识别；将识别结果发送给 LLM 并获取流式响应；将 LLM 输出流式送给 TTS 合成语音；同时监听用户打断信号。Pipeline 需要处理各种边界情况，如识别失败、LLM 超时、TTS 服务不可用等，并为每种情况提供优雅的降级策略。

**状态机管理**：Pipeline 内部维护一个简单的状态机，定义以下状态——IDLE（空闲等待）、LISTENING（正在聆听）、PROCESSING（正在处理）、SPEAKING（正在回答）、INTERRUPTED（被打断）。状态转换由事件驱动，清晰的的状态管理是实现打断机制和多轮对话的基础。

#### 4.2.2 FastAPI 服务设计（api_server.py）

使用 FastAPI 构建后端服务，充分利用其异步特性和现代化设计。服务设计遵循以下原则：

**Lifespan 管理**：利用 FastAPI 的 lifespan 机制管理应用生命周期。在 startup 事件中初始化各 API 客户端（ASR、LLM、TTS）、加载配置、启动后台任务（如 VAD 监听）。在 shutdown 事件中优雅关闭连接、清理资源、保存用户状态。这种方式比传统的 startup/shutdown 函数更加 Pythonic 且易于维护。

**API 端点设计**：服务提供以下核心端点——POST /对话/开始对话，接收麦克风音频流；POST /对话/打断，打断当前对话；GET /对话/状态，获取当前状态；POST /音色/切换，切换音色模式；GET /音色/列表，获取可用音色列表。所有端点均支持 WebSocket，提供实时双向通信能力。

**CORS 和安全**：配置合理的 CORS 策略允许前端跨域访问，实现请求验证和限流保护防止滥用。

#### 4.2.3 语音克隆实现（voice_clone.py）

语音克隆模块负责管理和使用声音特征，实现多音色支持：

**声音特征提取**：当需要克隆某个声音时，模块调用本地或云端的特征提取服务，从参考音频中提取声学特征（如speaker embedding、f0 轮廓、频谱特征等）。推荐使用开源工具 Coqui TTS 的 XTTS 模型或微软的 YourTTS，这些模型在零样本声音克隆任务上表现良好。

**特征缓存管理**：为每个已加载的音色维护一个特征缓存，包含声学特征向量、推荐语速/音调参数等。缓存采用 LRU（最近最少使用）淘汰策略，内存中最多缓存 10 个音色特征，确保内存使用可控。

**音色切换流程**：当用户发出切换指令时，Pipeline 解析目标音色名称，调用 VoiceClone 模块加载对应特征，然后通知 TTS 模块使用新音色。整个切换过程设计为可中断的，任何时候都可以切换到新的音色。

## 六、实施建议与优化方向

### 6.1 实施优先级建议

按照 MVP（最小可行产品）的思路，建议分阶段实施：

**第一阶段（MVP）**：实现核心对话功能，包括阿里云 ASR → 百炼 LLM → 阿里云 TTS 的完整 Pipeline，支持流式响应和简单的打断机制，预设 3 种音色供切换。此阶段目标是在 2 周内完成可演示的原型。

**第二阶段（增强）**：引入自克隆功能，使用 Coqui XTTS 实现用户声音克隆，优化 Pipeline 延迟至 1 秒以内，完善状态指示和错误处理。此阶段目标是在第一阶段基础上增加 2 周开发时间。

**第三阶段（完善）**：开发完整的前端界面，实现 WebSocket 实时通信，优化 UX 细节，增加更多预设音色，引入预测性响应等高级功能。

### 6.2 关键技术挑战与应对

**挑战一：端到端延迟控制**。语音助手场景下，用户对延迟极为敏感，理想情况下应控制在 800ms 以内。应对策略包括：全链路流式处理、预建立连接池、关键路径优化、使用边缘节点等。

**挑战二：打断场景的上下文处理**。当用户在 LLM 生成回复过程中打断时，需要决定如何处理被打断的内容。应对策略是采用语义完整性检测，当检测到用户打断时，不立即丢弃未完成的内容，而是将已生成的片段作为上下文缓存，待新输入完成后合并处理。

**挑战三：语音克隆质量控制**。自克隆效果受录音质量影响较大，用户环境噪音、录音距离、录音设备等因素都可能导致克隆效果下降。应对策略包括：提供录音质量检测和指导、对于低质量克隆提供降级方案（如提示用户重新录音）、在界面清晰标注当前使用的是克隆音色。

### 6.3 未来扩展方向

**情感识别与合成**：在当前功能稳定后，可以引入情感识别模块，分析用户语音中的情感状态（如高兴、悲伤、愤怒），并让 LLM 生成相应情感倾向的回复，甚至在 TTS 输出时使用相应的情感音色。

**多语言实时翻译**：基于当前 Pipeline，增加实时翻译能力，实现“说中文听英文”或“说英文听中文”的跨国对话场景。

**知识库接入**：将产品与用户私有知识库或企业知识库对接，让语音助手能够回答专业领域问题，提升在垂直场景下的实用性。

### Phase 1: 基础设施与全双工通信 (The Skeleton)
**目标：** 搭建极简的前后端骨架，打通 WebSocket 音频流的上传与下发。这是整个系统的“大动脉”。

* **后端任务 (`server.py`)：**
  * 使用 FastAPI 建立 WebSocket 路由，监听 `0.0.0.0:8000`。
  * 编写一个“回音壁 (Echo)”逻辑：接收前端传来的二进制音频块 (audio chunk)，不做任何处理，直接原样 send 回去。
* **前端任务 (`/web/src/hooks/useAudio.ts`, `useWebSocket.ts`)：**
  * 使用 `AudioWorkletProcessor` 唤起麦克风，以 16kHz/单声道/Float32 格式，每 20ms 切片发送 WebSocket 二进制流。
  * 接收后端的二进制流，并通过 Web Audio API 进行解码播放。

> **✅ 验收测试 (Echo Test)：**
> 启动前后端，戴上耳机对着麦克风说话。如果你能以极低的延迟（<100ms）听到自己的回音，且声音没有杂音、卡顿，说明**音频双向流转和网络穿透**测试通过。

### Phase 2: “耳朵”与端点检测 (ASR & VAD)
**目标：** 赋予系统听觉，并将原始音频转化为文本，同时能精准判断用户何时开口、何时闭嘴。

* **前端任务 (`/web/src/hooks/useVAD.ts`)：**
  * 引入 `Silero-VAD`。当检测到说话（Voice Active），开始向后端发送音频；当检测到静音（Voice Inactive），停止发送并发送一个 `{"type": "speech_end"}` 信号。
* **后端任务 (`/voxapi/core/asr_api.py`, `pipeline.py`)：**
  * 封装流式 ASR API（如阿里云 Paraformer）。
  * 将收到的 WebSocket 音频流喂给 ASR，提取中间结果 (`asr_partial`) 和最终结果 (`asr_final`)，通过 WebSocket 推送给前端。
* **UI 任务 (`/web/src/components/Subtitle.tsx`)：**
  * 极简字幕组件渲染：实时展示 ASR 传回的文字。

> **✅ 验收测试 (ASR Test)：**
> 对麦克风说话。前端不再播放回音，而是只在 VAD 激活时亮起指示灯，屏幕上的极简字幕能顺滑地流式打字上屏。停止说话后，后端能正确打印出最终的识别文本。测试通过。

### Phase 3: “大脑”与流式切片 (LLM & Routing)
**目标：** 接入大模型，实现意图识别（切换音色）和回复文本的流式切片。

* **后端任务 (`/voxapi/core/llm_api.py`, `pipeline.py`)：**
  * 封装大模型流式调用。
  * **核心逻辑**：拦截大模型输出的前几个 token，正则解析 `[VOICE_CTRL: {"target": "..."}]`。
  * 维护一个 buffer，按照标点符号（，。！？）对后续的文本进行切片，切出一段就 yield 一段。
* **打通管线：** ASR 的 `asr_final` 结果直接作为 LLM 的输入。

> **✅ 验收测试 (Brain Test)：**
> 说一句：“请用晓燕的声音回答，今天天气怎么样？”
> 后端终端应该清晰打印出两部分：
> 1. 成功提取 Target: `xiaoyan`
> 2. 流式分句打印回复文本（如：“今天天气很好，”、“适合出门。”）。
> 屏幕字幕同步显示流式回复。测试通过。

### Phase 4: “嘴巴”与秒切音色 (TTS & Voice Cache)
**目标：** 完成文字到声音的转换，并利用 Lifespan 实现零延迟的音色预热。

* **后端任务 (`/voxapi/core/tts_api.py`, `server.py`)：**

  * 将 Phase 3 中 LLM 切好的文本片段，连同提取好的 Target Embedding 一并送入流式 TTS 引擎。
  * TTS 吐出的 PCM 音频块通过 WebSocket 下发给前端。

> **✅ 验收测试 (Speaking Test)：**

### Phase 5: 核心灵魂：全双工打断 (The Interruption)
**目标：** 实现随时插话，打断系统当前的生成和播放。这是工程逻辑最复杂的一环。

* **前端任务：**
  * 如果当前系统正在播放音频，而 VAD 检测到你发出声音（持续 > 200ms），立刻发送控制帧 `{"action": "interrupt"}`。
* **后端任务 (`pipeline.py`)：**
  * 接收到 `interrupt` 后，立刻执行 `asyncio.Task.cancel()` 杀掉当前对应的 LLM 和 TTS 任务。
  * 清空待发送的音频队列。
  * 下发 `{"type": "stop_audio"}` 帧让前端掐断正在播放的音频。
* **克隆自己：** 前端维护 5s 环形录音缓存，在需要时作为 `speaker_audio` 发给后端，后端提取特征并缓存。

> **✅ 验收测试 (Barge-in Test)：**
> 让模型讲一个长故事。在它讲得正起劲时，你突然出声：“停，换个话题”。
> 系统的声音必须**瞬间停止**，并开始回应你的新话题。测试“克隆我的声音”指令，系统能使用你刚刚打断它的声音进行回复。测试通过。

# 技术路线  phase2 +
现在，项目正处于从“感知”向“认知与表达”过渡的关键节点。随着 Phase 3（LLM 大脑）和 Phase 4（TTS 嘴巴）的加入，数据流向会变得异常复杂（ASR -> LLM -> TTS -> 前端）。

审查了我们目前的工程结构和你的进度后，为了保证 nene 系统的极简风格和高扩展性，现在的当务之急是进行一次架构上的“收口”与重构，然后稳步推进 Phase 3。

🏗️ 当前工程结构的审查与优化建议
在 Phase 2 中，我们在 server.py 里直接实例化了 StreamASR 并处理了 WebSocket 的收发。但接下来如果不做调整，server.py 会迅速膨胀成“面条代码”。我们需要让各模块各司其职。

1. 后端：引入中枢神经 pipeline.py (状态机编排)
我们需要激活设计蓝图中的 pipeline.py。它将作为整个对话的中枢控制器。

server.py 的职责退化：只负责维护 WebSocket 连接、接收网络数据、处理断连异常。它把收到的所有有效负载（音频字节或 JSON 控制帧）直接扔给 pipeline.py。

pipeline.py 的职责确立：维护会话状态机（IDLE, LISTENING, THINKING, SPEAKING）。它负责监听 ASR 队列，一旦拿到 asr_final，立刻去调用 Qwen 引擎（llm_api.py），并对 Qwen 返回的流式 Token 进行标点切片，再送给 TTS。

2. 前端：Hooks 与组件分离 (极简代码美学)
目前 App.tsx 承担了 UI 渲染、WebSocket 管理、VAD 逻辑和音频缓冲。为了保持代码的清爽，我们需要将其拆解：

/web/src/hooks/useVADRecorder.ts：专门封装 audio-processor.js 的调用、麦克风权限和录音推流逻辑。

/web/src/hooks/useWebSocket.ts：专注网络通信和重连机制。

/web/src/components/Subtitle.tsx：负责呈现无冗余的极简字幕动效。

🚀 下一步：Phase 3 推进计划
在进行上述轻量重构的同时，我们将直接并入 Phase 3 的核心任务：接入 Qwen 引擎，实现流式切片。

具体的执行拆解：
Step 1: 编写大模型封装 (voxapi/core/llm_api.py)

构建一个异步生成器函数。它接收 ASR 传来的文本，向 Qwen 发起流式请求 (stream=True)。

核心逻辑：实现“标点切片缓冲”。它不能把 token 逐个往外抛，而是要缓存 token，直到遇到 [。，！？；] 等断句符号时，才把这一整句 yield 出来。这种粒度是下游 TTS 合成最自然的节奏。

意图拦截：检查输出的首个分句，用正则提取类似 [VOICE_CTRL: {"target": "xxx"}] 的学术级结构化控制指令，将其从正文中剥离。

Step 2: 编写编排器 (voxapi/pipeline.py)

创建一个 SessionPipeline 类，持有 StreamASR 和 LLMEngine 的实例。

实现事件循环串联：on_asr_final_received 触发 LLM 请求，LLM yield 出的文本切片通过 WebSocket 推回前端（Phase 3 暂时先推文本，Phase 4 再推给 TTS）。

Step 3: 净化 server.py

将繁杂的逻辑抽离，让 server.py 重新变得极其干净，只负责实例化 SessionPipeline 和转发 WebSocket 消息。

Step 4: Phase 3 验收测试

前端保持不变。对着麦克风说：“请用极简风格介绍一下武汉”。

期望结果：前端先显示你的问题，随后，屏幕上的字幕开始一段一段地（按标点符号粒度）流式涌现出 Qwen 的回答，而不是像打字机一样一个字一个字蹦。
